# Storage

## Назначение

Сервис хранения транскодированных треков. Принимает сырое аудио, транскодит (ffmpeg), кладёт в backend (local FS или S3), отдаёт по ссылке.

## Роль в архитектуре

Storage — единственная точка, где аудио физически живёт и раздаётся. Но **разные потребители получают разные URL**:

- **Desktop-клиенты** — всегда идут **через storage** (`http://storage/...`). S3 URL наружу никогда не светится. Причины: контроль доступа, ротация бэкенда (локальный FS ↔ S3 без изменений в клиенте), единый origin для CDN/прокси на фронте.
- **Worker (AI)** — получает **стабильный URL на storage** (`{storage}/redirect/hq/{file}.ogg`). На S3-бэкенде storage на каждый хит пересчитывает свежий presigned и отвечает 307 → воркер следует по редиректу и качает байты **напрямую с S3** (не через storage). На local-бэкенде storage сам отдаёт байты по тому же URL. Причины: URL не протухает (worker-задача может ждать минуты/часы), воркер не держит в памяти presigned с ограниченным TTL, локальный режим не требует S3.

## Правила

- **Клиенты не должны знать про S3.** Никогда не отдавать S3 URL в desktop-ответах. Только свой origin storage.
- **Воркер получает только `{storage}/redirect/...` URL.** Свежий presigned генерируется storage'ом на каждый запрос, поэтому URL можно кэшировать сколько угодно.
- **Бэкенд хранилища абстрагирован** (`backend/local.rs`, `backend/s3.rs`). Переключение — через `STORAGE_BACKEND` env. `/redirect/` работает на обоих.
