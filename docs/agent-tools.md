# Гайд по тулзам OpenCode (для агента)

Окружение OpenCode использует **camelCase** имена параметров и не всегда совпадает с системным промптом Claude.
Здесь — то, что реально работает в этом проекте. Не тратить токены на эксперименты.
**И ТУТ ВСЕ ПРАВИЛЬНО ПО РЕГИСТРУ, НЕ ПЫТАЙСЯ СТРОИТЬ ИЗ СЕБЯ САМОГО УМНОГО, КАК ТУТ НАПИСАНО ТАК И ВЫЗЫВАЙ**
**ЧИСЛО — ЧИСЛОМ, СТРОКА — СТРОКОЙ**

Специфика `edit_file` вынесена отдельно: см. [`agent-edit-tool.md`](./agent-edit-tool.md).

## Чтение файлов

### `view_file`

| Параметр       | Тип    | Назначение                              |
|----------------|--------|-----------------------------------------|
| `AbsolutePath` | string | Абсолютный путь (обязателен **первым**) |
| `StartLine`    | number | Опционально, 1-indexed                  |
| `EndLine`      | number | Опционально, 1-indexed                  |

- Без `StartLine`/`EndLine` — весь файл.
- Возвращает строки в формате `N: <содержимое>`. При копировании в `edit_file.oldString` префикс `N: ` отрезать.
- НЕ использовать `cat`/`head`/`tail` через `run_command`.

### `read_notebook` — для `.ipynb`. Не использовать `view_file` на ноутбуках.

## Поиск

### `grep_search` — по содержимому

| Параметр        | Тип      | Назначение                                              |
|-----------------|----------|---------------------------------------------------------|
| `SearchPath`    | string   | Абсолютный путь к директории или файлу (обязателен)     |
| `Query`         | string   | Regex по умолчанию; literal — через `FixedStrings:true` |
| `Includes`      | string[] | Glob-фильтры, напр. `["*.ts", "!**/node_modules/**"]`   |
| `MatchPerLine`  | boolean  | Показывать строки с матчем. Только для узких поисков    |
| `CaseSensitive` | boolean  | По умолчанию false                                      |

- НЕ запускать `rg`/`grep` через `run_command`.
- Для широких поисков `MatchPerLine: false` — иначе вывод раздуется.

### `find_by_name` — по имени/glob

Использовать вместо `find`/`ls -R`. Поддерживает `**` рекурсию кроссплатформенно.

### Когда использовать `code_search`/Task subagent

Для исследовательских вопросов («где обрабатываются ошибки клиента?», «как устроен модуль X?»).
Для конкретного файла/функции — `grep_search` + `view_file` напрямую.

## Команды

### `run_command`

| Параметр        | Тип     | Назначение                                              |
|-----------------|---------|---------------------------------------------------------|
| `CommandLine`   | string  | Команда как для shell (обязателен)                      |
| `description`   | string  | Короткое описание для UI (обязателен)                   |
| `Blocking`      | boolean | Опционально. true — ждать завершения. По умолчанию sync |
| `SafeToAutoRun` | boolean | Только для гарантированно безопасных команд (read-only) |

- Использовать только для git/cargo/pnpm/npm/docker и т.п. **Не** для чтения файлов и не для эха в чат.
- `cd <dir> && cmd` работает — для команд, которым нужен конкретный cwd (например `cargo build` в `backend/gateway`).
- НЕ устанавливать пакеты / не делать destructive ops без подтверждения пользователя.

## Запись/правка файлов

### `edit_file` — править существующие. См. [`agent-edit-tool.md`](./agent-edit-tool.md).

### `write_to_file` — создать новый файл

| Параметр      | Тип     | Назначение                              |
|---------------|---------|-----------------------------------------|
| `TargetFile`  | string  | Абсолютный путь (обязателен **первым**) |
| `CodeContent` | string  | Полное содержимое                       |
| `EmptyFile`   | boolean | true — создать пустой файл              |

- **Не использовать для перезаписи существующих файлов** — использовать `edit_file`.
- Перед созданием убедиться что файла нет (`view_file` или `find_by_name`).

### `edit_notebook` — для `.ipynb`. Удалять ячейки нельзя — попросить пользователя.

## Параллельность

- Независимые тулзы вызывать **в одном сообщении** (несколько `<invoke>` блоков).
- Зависимые (результат одной нужен для аргумента другой) — последовательно.
- Несколько `edit_file` к разным файлам — параллельно ок.
- Несколько `edit_file` к одному файлу с непересекающимися `oldString` — тоже ок, но безопаснее последовательно или через `multi_edit` (если доступен).

## Типичные грабли в этом окружении

| Симптом                                     | Причина                                                       |
|---------------------------------------------|---------------------------------------------------------------|
| `SchemaError(Missing key at ["filePath"])`  | Вызов `edit_file` с пустыми/неправильными ключами             |
| `SchemaError(Missing key at ["oldString"])` | Передал только `TargetFile`                                   |
| `Expected number, got "175"`                | `view_file.StartLine`/`EndLine` ожидает number, не string     |
| `could not find Cargo.toml`                 | Запустил `cargo build` из корня монорепо — нужен `cd` в крейт |
| `old_string not found`                      | Не совпали отступы/невидимые символы — перечитать `view_file` |
